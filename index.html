Screen Reader benefits

  • Use keyboard to navigate and ﬁnd content
  • 98.6% of screen reader users have JS enabled!
  • Highly customizable

HTML Counts

Headings

  • Main way screen reader users navigate
  • Do your main content areas have headings?
  • Are they descriptive?
  • Do they follow a hierarchy? (h1 >> h6)

Reader ignores

  • img with empty alt attribute alt=""
  • display: none;
  • visibility: hidden;
  • :before content, :after content (sort of)
  • keep in mind for icons and icon fonts!

Reader won't ignore

  • content “hidden” with opacity, z-index, height
  • oﬀ-screen positioning (text-indent, top, left)
  • CSS clipping
			.wcag-hidden {
				border: 0;
				clip: rect(0 0 0 0);
				height: 1px;
				margin: -1px;
				padding: 0;
				position: absolute;
				width: 1px;
			}

Beyond the basics

There’s gotta be more, right?

ARIA

  • Semantic information and better interactions for screen readers
  • Applied directly to HTML
  • Part of HTML5 spec
  • Roles, states & properties
  • Does not aﬀect styles or non-SR behavior

Roles

  • Create new semantic meaning for elements via “role-” attribute
  • Once set, they don’t change
  • <nav role="navigation">
  • <article role="article">
  • <div role="tablist">
  • <div role="combobox>

Widget Roles

  • Semantic meaning to your custom components
  • tooltip
  • slider
  • dialog
  • tab
  • menu
  • alert
  • and many more!

States & Properties

  • Describe relationships - between content & between user interactions
  • updated via JS on UI changes
  • attribute starts with “aria-” preﬁx
  • <section aria-labelledby="MainHeading">
  • <div aria-expanded="true">
  • etc.

Content Relationships

  • Semantically link labels to content or add them when missing
  • aria-labelledby, aria-label

Using ARIA Wisely

  • ARIA is a bridge, not a replacement.
  • USE plain HTML if you can
  • Not magic and makes no promises
  • Events, focus management, keyboard support, and meaningful structure is still up to you
  • Only way to know for sure... TEST
